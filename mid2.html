Attacks and Exploits

Short, focused slides. One idea per slide with plain-language explanation and classroom-safe examples.

DoS - What it is

A Denial of Service (DoS) attack makes a service slow or unavailable by exhausting a resource from a single source.

Short example: One client sends many requests until the server queue fills and responses become very slow.
Think of one person repeatedly calling a help desk until the phone lines are busy and others cannot get through.

DoS - Signs and defenses

  • Signs: one IP with unusually high request count, CPU spikes, and many 503/timeouts.
  • Quick defenses: rate limiting, request timeouts, and per-IP connection caps.

DDoS - What it is

Distributed DoS uses many devices at once (botnet traffic), so blocking one source does not stop the attack.

Short example: Thousands of small requests from many devices saturate bandwidth or exhaust server resources.
Like many people calling the same help desk at once from different phones.

DDoS - Defenses

  • Use CDN and cloud DDoS protection to absorb and filter traffic.
  • Autoscale and cache to reduce load on origin systems.
  • Apply traffic shaping, anomaly detection, and geo/IP filtering carefully.

SQL Injection - Simple idea

When an app mixes raw user input into SQL strings, attackers can change query logic.

Analogy: If you paste user words into a command sentence, the user can add extra command parts.
Treat user input as data only, never as query language.

SQLi - Vulnerable code (exact)

// Vulnerable PHP (classroom demo only)
$username = $_POST['username'];
$password = $_POST['password'];

$sql = "SELECT id, name FROM users WHERE username = '" . $username . "' AND password = '" . $password . "';";
// $result = $db->query($sql);
Input is concatenated into SQL, so special characters can alter query behavior.

SQLi - Exact input and result

Inputs:
username: alice
password: ' OR '1'='1
Executed SQL: 
SELECT id, name FROM users WHERE username = 'alice' AND password = '' OR '1'='1';
'1'='1' is always true, so authentication logic can be bypassed.

SQLi - Safe fix (prepared statements)

// Safe PHP with PDO
$stmt = $pdo->prepare('SELECT id, name FROM users WHERE username = :u AND password = :p');
$stmt->execute([':u' => $_POST['username'], ':p' => $_POST['password']]);
$user = $stmt->fetch();
Prepared statements separate code from data, so input cannot rewrite SQL structure.

XSS - Exact classroom example

Cross-Site Scripting (XSS) occurs when untrusted input is inserted into the page as HTML/JS instead of plain text.

Vulnerable code 
// Vulnerable JavaScript
const q = new URLSearchParams(location.search).get('q') || '';
document.getElementById('searchResult').innerHTML = q;
Exact input and rendered result 
Input (URL query):
?q=%3Cimg%20src=x%20onerror=alert('XSS')%3E

Rendered HTML:
<img src=x onerror=alert('XSS')>
Safe fix 
// Safe output handling
const q = new URLSearchParams(location.search).get('q') || '';
document.getElementById('searchResult').textContent = q;
Teach output encoding, CSP, and HttpOnly cookies. Keep payloads harmless in class demos (for example, alert only).

IP Spoofing and Ransomware - Short notes

IP Spoofing:
  • Attacker forges source IP addresses in packets.
  • Risk: breaks naive IP trust and supports reflection/amplification abuse.
  • Defenses: anti-spoofing (BCP 38) and strong app-layer authentication.
Ransomware (worm-like, conceptual):
  • Combines exploit behavior with file encryption and lateral spread attempts.
  • Teaching focus: detection, isolation, recovery playbooks, backups, and segmentation.

Summary and safe classroom activities

  • Keep demos short and isolated: one concept per task.
  • Use VMs or host-only networks, never production systems.
  • Use intentionally vulnerable training apps (for example, OWASP Juice Shop).
  • Practice defense workflow: monitoring, patching, segmentation, and backups.
