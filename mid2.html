Attacks & Exploits

Short, focused slides. Each slide covers one idea with a clear example and plain-language explanation.

DoS — What it is

A Denial of Service (DoS) attack makes a service slow or unavailable by exhausting a resource from a single source.

Short example: One client sends many requests until the server's request queue fills and responses slow down.
Plain explanation: Think of a single person repeatedly calling a help desk until the phone lines are busy and others can't get through.

DoS — Signs & Defenses

  • Signs: one IP with huge request count; CPU or memory spikes; many 503/timeouts.
  • Quick defenses: rate limiting, connection timeouts, per-IP caps.

DDoS — What it is

Distributed DoS uses many devices at once (a botnet) so blocking a single IP won't stop the attack.

Short example: Thousands of small requests from many devices together saturate bandwidth or exhaust server resources.
Plain explanation: Like many people calling a help desk at once from different phones — blocking one phone doesn't help.

DDoS — Defenses

  • Use CDN and cloud DDoS protection to absorb traffic.
  • Autoscale and cache to reduce load on origin servers.
  • Filter suspicious traffic patterns and apply geo/IP rules when appropriate.

SQL Injection — Simple idea

When an application mixes user input directly into a database query, an attacker can change the query's meaning.

Analogy: If you build a sentence by pasting user text into it, a user could insert extra sentence parts that change the meaning.
Plain explanation: Treat user input as data only — never as part of the query language.

SQLi — Vulnerable code (classroom)

// Vulnerable PHP (for sandboxed demo only)
$username = $_POST['username'];   // user input
$password = $_POST['password'];   // user input

// Unsafe: concatenates input directly into SQL
$sql = "SELECT id, name FROM users WHERE username = '" . $username . "' AND password = '" . $password . "';";
// $result = $db->query($sql);
This code builds SQL by pasting user text into the query. If the input contains SQL syntax, it can change the query's logic.

SQLi — Exact input and what happens

Inputs (example):
username: alice
password: ' OR '1'='1
The database sees: 
SELECT id, name FROM users WHERE username = 'alice' AND password = '' OR '1'='1';
Because '1'='1' is always true, the WHERE clause may match rows even without the real password.

SQLi — Safe fix (prepared statements)

// Safe PHP using PDO prepared statements
$stmt = $pdo->prepare('SELECT id, name FROM users WHERE username = :u AND password = :p');
$stmt->execute([':u' => $_POST['username'], ':p' => $_POST['password']]);
$user = $stmt->fetch();
Prepared statements keep code and data separate. The database treats inputs as values, not SQL code, so special characters cannot change query structure.

XSS — Plain explanation

Cross-Site Scripting (XSS) lets an attacker place script code into pages other users view. That script runs in the victim's browser and can steal session data or perform actions as the user.

Class demo idea: Show a page that prints a search term raw vs encoded. Use a harmless example to show the difference between text and executable script.
Teach output encoding and Content Security Policy (CSP) as defenses. Use HttpOnly cookies to reduce theft risk.

IP Spoofing & Ransomware — Short notes

IP Spoofing:
  • Attacker fakes source IP in packets.
  • Risk: breaks IP-based trust; used in reflection/amplification attacks.
  • Defenses: anti-spoofing (BCP 38), strong app-layer auth (TLS, tokens).
Ransomware (worm-like) — conceptual:
  • Combines exploit + payload that encrypts files and tries to spread.
  • Focus teaching on detection, containment, and recovery (backups, segmentation).

Summary & Safe Classroom Activities

  • Keep demos short and focused: one concept per slide.
  • Always use isolated VMs or host-only networks for hands-on labs.
  • Use intentionally vulnerable apps (OWASP Juice Shop) for learning, not production systems.
  • Practice detection and recovery: monitoring, patching, segmentation, and backups.
